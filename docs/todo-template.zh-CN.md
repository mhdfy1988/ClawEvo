# TODO.md 模板

这是一份统一的 `todo.md` 模板，后续新的项目 TODO、阶段 TODO、专项 TODO 都建议按这个格式来写。

核心原则：

- 顶部先写标题和一句话说明
- 正文固定分成 `待办 / 进行中 / 已完成`
- 每条主任务尽量带上 `预估时间 / 标签 / 负责人 / 日期`
- 子任务用复选框缩进展开
- 状态流转时直接移动到对应分区，不要同一条任务同时出现在多个分区

---

## 模板正文

```markdown
# <项目或阶段名称> TODO

<一句话说明这份 TODO 用来管理什么>

## 待办

- [ ] <任务标题> ~<预估工期> #<标签> @<负责人> <YYYY-MM-DD>
  - [ ] <子任务 1>
  - [ ] <子任务 2>

- [ ] <任务标题> ~<预估工期> #<标签> @<负责人> <YYYY-MM-DD>
  - [ ] <子任务 1>
  - [ ] <子任务 2>

## 进行中

- [ ] <任务标题> ~<预估工期> #<标签> @<负责人> <YYYY-MM-DD>
  - [ ] <子任务 1>
  - [ ] <子任务 2>

## 已完成

- [x] <任务标题> #<标签> @<负责人> <YYYY-MM-DD>
  - [x] <关键子任务或里程碑>

- [x] <任务标题> #<标签> @<负责人> <YYYY-MM-DD>
```

---

## 字段约定

- `<任务标题>`
  说明: 用一句短话写清楚任务本身，避免写成大段描述。
- `~<预估工期>`
  说明: 可选，建议写成 `~1d`、`~3d`、`~1w` 这种形式。
- `#<标签>`
  说明: 可选，建议标任务类别，例如 `#后端`、`#文档`、`#测试`、`#架构`。
- `@<负责人>`
  说明: 可选，如果是个人 TODO 也可以先写自己或留空。
- `<YYYY-MM-DD>`
  说明: 建议写目标日期或完成日期。

---

## 推荐用法

### 1. 新任务

新任务统一放进 `待办`：

```markdown
- [ ] 补 tool result explain ~2d #后端 @自己 2026-03-15
  - [ ] 输出 policyId
  - [ ] 输出 droppedSections
  - [ ] 输出 artifact 回查路径
```

### 2. 开始执行

开始做时，把整条任务移动到 `进行中`，不要复制一份。

### 3. 完成后

完成后移动到 `已完成`，主任务和已完成子项统一改成 `[x]`：

```markdown
- [x] 补 tool result explain #后端 @自己 2026-03-15
  - [x] 输出 policyId
  - [x] 输出 droppedSections
  - [x] 输出 artifact 回查路径
```

## 完整示例

```markdown
# 阶段 2 收尾 TODO

本清单用于跟踪阶段 2 剩余收尾工作。

## 待办

- [ ] 补 artifact sidecar 落盘闭环 ~3d #后端 @自己 2026-03-18
  - [ ] 增加 artifact store
  - [ ] 落正文到外置文件
  - [ ] explain 输出回查路径

- [ ] 补阶段 2 指标与出口验收 ~2d #文档 @自己 2026-03-20
  - [ ] 整理压缩率指标
  - [ ] 整理 explain 覆盖情况
  - [ ] 输出阶段总结

## 进行中

- [ ] 补 tool_result_persist 裁剪原因 explain ~2d #后端 @自己 2026-03-15
  - [ ] 输出 policyId
  - [ ] 输出 droppedSections
  - [ ] 输出 artifact/sourcePath/sourceUrl

## 已完成

- [x] 整理阶段 2 收尾 TODO #文档 @自己 2026-03-13

- [x] 补文档总览索引 #文档 @自己 2026-03-13
```

---

## 建议

- 阶段型工作用这一套模板最合适
- 如果任务很多，优先保持“看一眼就知道当前在哪做什么”
- 如果需要额外说明，放在标题下面的一句话简介里，不要把正文写成大段散文
